# Objective

Remove `'unsafe-inline'` and `'unsafe-eval'` from the Content-Security-Policy `script-src` directive. Replace with nonce-based CSP using Next.js App Router's built-in nonce support. This addresses NEW-001 — the current CSP defeats its own purpose by allowing arbitrary inline scripts.

# Findings Addressed

- **NEW-001** (Medium): CSP uses `'unsafe-inline'` + `'unsafe-eval'` on script-src

# Files to Update

| File | Change |
|------|--------|
| `frontend/next.config.mjs` | Remove `'unsafe-inline'` and `'unsafe-eval'` from CSP header, add `'nonce-{nonce}'` placeholder |
| `frontend/src/middleware.ts` | Generate per-request CSP nonce, set header |
| `frontend/src/app/layout.tsx` | Pass nonce to Script components if needed |

# Implementation Plan

1. Research Next.js 14+ nonce-based CSP pattern (middleware generates nonce, injects into CSP header)
2. Write failing tests for middleware nonce generation
3. Create/update middleware to generate crypto nonce per request and set CSP header with nonce
4. Update `next.config.mjs` to remove static CSP header (middleware handles it dynamically)
5. Remove `'unsafe-eval'` — Next.js production builds don't need it
6. Keep `'unsafe-inline'` only in `style-src` if CSS modules require it (evaluate)
7. Verify no inline scripts break in dev and production builds

# Acceptance Criteria

- [x] `script-src` does NOT contain `'unsafe-inline'` or `'unsafe-eval'`
- [x] CSP header uses nonce-based approach (`'nonce-<value>'`)
- [x] Nonce is unique per request (crypto random)
- [x] Next.js pages render without CSP violations in dev and build
- [x] Existing CSP tests updated to reflect new policy
- [x] `next build` succeeds without errors
