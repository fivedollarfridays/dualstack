# Objective

Harden the frontend's direct database layer (`src/db/`) to prevent unaudited writes and validate token configuration. The layer exists for SSR read performance but currently holds a full-access token and has no technical write prevention.

# Findings Addressed

- **NEW-003** (Medium): Frontend DB layer with full-access token creates unaudited read path
- **NEW-004** (Low): TURSO_AUTH_TOKEN guard accepts empty string, giving false confidence
- **AUDIT-010** (Medium, partial): Dual write path concern still active

# Files to Update

| File | Change |
|------|--------|
| `frontend/src/db/index.ts` | Validate token is non-empty when URL is remote; add read-only documentation |
| `frontend/.env.example` | Add comment noting token should be read-only for frontend |
| `docs/ARCHITECTURE.md` | Update dual-write path section with enforcement guidance |

# Implementation Plan

1. Write failing test for token validation (empty string on remote URL should throw)
2. Update `frontend/src/db/index.ts` to validate `TURSO_AUTH_TOKEN` is non-empty when `TURSO_DATABASE_URL` is not a `file:` URL
3. Add documented convention to prevent write imports from `src/db/`
4. Update `.env.example` with guidance that frontend token should be read-only scoped
5. Update `docs/ARCHITECTURE.md` with enforcement recommendation (Turso read-only tokens)

# Acceptance Criteria

- [x] Empty `TURSO_AUTH_TOKEN` with remote `TURSO_DATABASE_URL` throws clear error at startup
- [x] Local file URLs (dev) work without a token
- [x] `.env.example` documents read-only token requirement for frontend
- [x] `docs/ARCHITECTURE.md` updated with Turso read-only token guidance
- [x] Existing frontend tests still pass
