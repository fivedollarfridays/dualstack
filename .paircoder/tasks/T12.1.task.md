# Objective

Backend SQLAlchemy cannot connect to Turso (libsql:// protocol). The current `database.py` falls back to local SQLite with a `RuntimeError` in production, meaning the backend has **no working remote database connection**. This task evaluates two options and implements the chosen path:

1. **Option A -- sqlalchemy-libsql dialect**: Install `sqlalchemy-libsql`, update the engine creation to handle `libsql://` URLs natively, and keep Turso as the production database.
2. **Option B -- PostgreSQL migration**: Replace Turso/libsql with PostgreSQL (e.g., Neon, Supabase, or self-hosted), update SQLAlchemy connection strings, and migrate the schema.

The decision must be documented with rationale before implementation begins.

# Files to Update

- `backend/app/core/database.py` -- Remove/resolve the `RuntimeError` for `libsql://`, update engine creation for the chosen driver
- `backend/requirements.txt` or `pyproject.toml` -- Add `sqlalchemy-libsql` (Option A) or `psycopg2-binary`/`asyncpg` (Option B)
- `backend/alembic/env.py` -- Update Alembic config to work with the chosen database driver
- `backend/alembic/versions/` -- Generate and verify a migration if schema changes are needed
- `docs/ARCHITECTURE.md` -- Document the database decision and connection architecture
- `.env.example` -- Update example environment variables for the chosen database URL format

# Implementation Plan

## Phase 1: Evaluation (no code changes)

1. Read `backend/app/core/database.py` and identify the exact `RuntimeError` and fallback logic
2. Check if `sqlalchemy-libsql` supports the project's SQLAlchemy version (2.x required)
3. Test `sqlalchemy-libsql` connectivity to a Turso database in a local sandbox
4. Evaluate PostgreSQL as an alternative: provisioning complexity, cost, migration effort
5. Write a short decision record (2-3 paragraphs) in `docs/ARCHITECTURE.md` under a "Database Decision" heading

## Phase 2: Implementation (chosen path)

6. Install the chosen driver dependency
7. Update `database.py` engine creation:
   - Remove the `RuntimeError` / fallback block
   - Add proper URL parsing for the chosen protocol
   - Ensure `create_engine` / `create_async_engine` works with the new driver
8. Update Alembic `env.py` to use the same engine/URL logic
9. Run existing Alembic migrations against the new database to verify compatibility
10. Update `.env.example` with the correct `DATABASE_URL` format

## Phase 3: Verification

11. Write tests for database connection (connection succeeds, session creates, basic CRUD)
12. Run the full backend test suite to confirm no regressions
13. Verify Alembic `upgrade head` and `downgrade` work cleanly
14. Test with a real remote database URL (not just SQLite fallback)
