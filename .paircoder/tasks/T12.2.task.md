# Objective

The frontend contains a Drizzle ORM layer at `frontend/src/db/` that is not used by any page for SSR data reads. This creates two problems:

1. **Dual-write risk**: Two ORM layers (backend SQLAlchemy + frontend Drizzle) could write to the same database, creating race conditions and data inconsistency.
2. **Schema sync burden**: Schema definitions must be maintained in two places (Alembic models + Drizzle schema), increasing the chance of drift.

This task requires a deliberate decision: either **remove** the frontend DB layer entirely (recommended if all data access goes through the backend API), or **justify and implement** actual SSR reads that demonstrate a legitimate use case.

# Files to Update

### If removing (recommended path):

- `frontend/src/db/` -- Delete entire directory (schema, client, migrations config)
- `frontend/src/app/` -- Remove any imports referencing `src/db/`
- `frontend/package.json` -- Remove `drizzle-orm`, `drizzle-kit`, `@libsql/client` dependencies and related scripts
- `frontend/drizzle.config.ts` -- Delete if it exists
- `frontend/tsconfig.json` -- Remove path aliases for `@/db` if configured
- `docs/ARCHITECTURE.md` -- Document the decision to use backend-only data access

### If justifying (alternative path):

- `frontend/src/db/` -- Keep and clean up
- `frontend/src/app/` -- Add at least one page with actual SSR database reads
- `docs/ARCHITECTURE.md` -- Document why both layers exist, which is authoritative, and the data flow

# Implementation Plan

## Phase 1: Audit current usage

1. List all files in `frontend/src/db/` and understand what schema/tables are defined
2. Search the entire frontend codebase for imports from `src/db/` or `@/db`
3. Check if any API route or page component performs direct database reads
4. Check `frontend/package.json` for drizzle-related dependencies and scripts
5. Document findings: is the DB layer used anywhere? If so, where?

## Phase 2: Decision

6. If no usage found: proceed with removal
7. If usage found: evaluate whether it is justified or should be replaced with API calls
8. If a valid SSR use case exists: implement it properly with documentation

## Phase 3A: Removal (if chosen)

8. Delete `frontend/src/db/` directory
9. Remove drizzle-related dependencies from `package.json` (`drizzle-orm`, `drizzle-kit`, `@libsql/client`)
10. Remove drizzle scripts from `package.json` (e.g., `db:push`, `db:generate`, `db:studio`)
11. Delete `frontend/drizzle.config.ts` if present
12. Search for and fix any broken imports referencing the deleted directory
13. Run `npm run build` to verify no compilation errors
14. Update `docs/ARCHITECTURE.md` with the decision

## Phase 3B: Justification (if chosen)

8. Implement at least one SSR read in a Next.js page or API route
9. Add tests for the SSR read
10. Document the data flow and ownership in `docs/ARCHITECTURE.md`
11. Define clearly which layer (backend vs frontend) is authoritative for writes
