# Objective

Two migration systems currently manage the same database:

- **Alembic** (backend, Python) -- manages SQLAlchemy models
- **Drizzle Kit** (frontend, TypeScript) -- manages Drizzle ORM schema

Having two migration systems for one database creates risks: conflicting migrations, schema drift, and unclear ownership of database changes. This task consolidates to a single authoritative migration system.

The outcome depends on T12.2:
- **If T12.2 removed the frontend DB layer**: Remove Drizzle migrations and make Alembic the sole migration system.
- **If T12.2 kept the frontend DB layer**: Define which system is authoritative, document the boundary, and ensure no overlap.

# Files to Update

### If Drizzle is removed (likely, follows T12.2 removal):

- `frontend/drizzle/` -- Delete migration directory
- `frontend/drizzle.config.ts` -- Delete configuration file
- `frontend/package.json` -- Remove `drizzle-kit` scripts (`db:generate`, `db:migrate`, `db:push`, `db:studio`)
- `backend/alembic/` -- Verify all tables are covered by Alembic migrations
- `docs/ARCHITECTURE.md` -- Document Alembic as the single migration system

### If both are kept (follows T12.2 justification):

- `docs/ARCHITECTURE.md` -- Document which tables belong to which system
- `backend/alembic/` -- Ensure no overlap with Drizzle-managed tables
- `frontend/drizzle/` -- Ensure no overlap with Alembic-managed tables
- Add a CI check or script that validates no table is managed by both systems
