# Objective

No user-to-Stripe-customer mapping exists in the codebase. The billing module (`backend/app/billing/`) can create Stripe Checkout sessions but has no way to link a Clerk `user_id` to a Stripe `customer_id`. This blocks the billing portal endpoint (currently returns 501) and any subscription-state storage.

Create a new `users` module (`backend/app/users/`) following the same hub-and-spoke pattern as the existing `items` module. The module provides a SQLAlchemy model that stores the Clerk user ID alongside the Stripe customer/subscription mapping, plus a service layer for CRUD operations and an Alembic migration.

# Background

- The existing `Item` model in `backend/app/items/models.py` uses `String` primary keys (UUIDs), `DateTime` columns with `func.now()` defaults, and the shared `Base` from `app.core.database`.
- The `get_current_user_id` dependency in `backend/app/core/auth.py` returns a Clerk `sub` claim (string).
- The billing portal route at `backend/app/billing/routes.py:34-52` explicitly documents that it needs a user->customer mapping table.
- The billing service at `backend/app/billing/service.py:28-40` has a `create_portal_session` function ready to accept a `customer_id`.
- Alembic is configured at `backend/alembic/` with `backend/alembic.ini`.

# Implementation Plan

## 1. Create the Users model (`backend/app/users/models.py`)

Define a `User` SQLAlchemy model mapped to a `users` table:

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | `String` | PK, default UUID | Internal row ID |
| `clerk_user_id` | `String` | unique, not null, indexed | From Clerk JWT `sub` claim |
| `stripe_customer_id` | `String(255)` | nullable, unique, indexed | Set on first checkout |
| `subscription_plan` | `String(50)` | nullable, default `"free"` | e.g. `free`, `pro`, `enterprise` |
| `subscription_status` | `String(50)` | nullable, default `"none"` | e.g. `none`, `active`, `past_due`, `canceled` |
| `created_at` | `DateTime` | server_default `func.now()` | Row creation timestamp |
| `updated_at` | `DateTime` | server_default `func.now()`, onupdate `func.now()` | Last modification |

Follow the same pattern as `backend/app/items/models.py` (uuid default, Column-based declarations).

## 2. Create Pydantic schemas (`backend/app/users/schemas.py`)

- `UserResponse` -- read schema with `from_attributes = True`
- `UserCreate` -- only `clerk_user_id` required; everything else has defaults
- `UserUpdate` -- all fields optional (for partial updates to subscription fields)

## 3. Create service layer (`backend/app/users/service.py`)

Async functions operating on `AsyncSession`:

- `get_or_create_user(db, clerk_user_id) -> User` -- idempotent; returns existing or creates new
- `get_user_by_clerk_id(db, clerk_user_id) -> User | None` -- lookup
- `get_user_by_stripe_id(db, stripe_customer_id) -> User | None` -- lookup
- `update_user(db, clerk_user_id, data: UserUpdate) -> User` -- partial update
- `link_stripe_customer(db, clerk_user_id, stripe_customer_id) -> User` -- sets stripe_customer_id

## 4. Create `__init__.py` (`backend/app/users/__init__.py`)

Hub file re-exporting public API: `User`, `UserResponse`, `UserCreate`, `UserUpdate`, and service functions.

## 5. Generate Alembic migration

```bash
cd backend && alembic revision --autogenerate -m "add users table with stripe mapping"
```

Verify the generated migration creates the `users` table with all columns and indexes.

## 6. Write tests

- `backend/tests/test_users/test_user_model.py` -- model instantiation, defaults, constraints
- `backend/tests/test_users/test_user_service.py` -- CRUD operations with in-memory SQLite
