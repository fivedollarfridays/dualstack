# Objective

The Stripe webhook handler at `backend/app/billing/service.py:handle_webhook` currently verifies the signature and logs audit events, but does not update any database state. It returns `{"handled": True}` for known event types without persisting anything.

Implement database-backed handlers for three critical Stripe webhook events so that the `users` table (created in T13.1) stays in sync with Stripe subscription state:

1. **`checkout.session.completed`** -- Link the Stripe customer ID to the Clerk user and set initial subscription status
2. **`customer.subscription.updated`** -- Update plan name and status when subscription changes (upgrade, downgrade, payment issue)
3. **`customer.subscription.deleted`** -- Mark subscription as canceled

# Background

- The existing `handle_webhook` function in `backend/app/billing/service.py:57-85` dispatches by event type using `_WEBHOOK_ACTIONS` dict. Currently only logs audit events.
- The `checkout.session.completed` event payload contains `metadata.user_id` (set at checkout creation, line 21 of service.py) and `customer` (Stripe customer ID).
- The `customer.subscription.updated/deleted` events contain `customer` (Stripe customer ID) and subscription details (`items.data[0].price.lookup_key` or `items.data[0].price.id`, `status`).
- The Users service from T13.1 provides `get_or_create_user`, `link_stripe_customer`, and `update_user`.
- The database session is available via `app.core.database.get_db()` (async generator dependency).

# Implementation Plan

## 1. Create billing event service (`backend/app/billing/webhook_handlers.py`)

Extract webhook processing into a dedicated module to keep `service.py` focused:

- `handle_checkout_completed(db, event_data) -> None`
  - Extract `user_id` from `session.metadata.user_id`
  - Extract `customer_id` from `session.customer`
  - Extract subscription ID from `session.subscription`
  - Call `get_or_create_user(db, clerk_user_id=user_id)`
  - Call `link_stripe_customer(db, clerk_user_id=user_id, stripe_customer_id=customer_id)`
  - Set initial `subscription_status="active"` and determine plan from line items

- `handle_subscription_updated(db, event_data) -> None`
  - Extract `customer_id` from `subscription.customer`
  - Look up user by `stripe_customer_id`
  - Update `subscription_status` from `subscription.status` (map Stripe statuses: `active`, `past_due`, `unpaid`, `trialing`, etc.)
  - Update `subscription_plan` from price lookup key or price ID

- `handle_subscription_deleted(db, event_data) -> None`
  - Extract `customer_id` from `subscription.customer`
  - Look up user by `stripe_customer_id`
  - Set `subscription_status="canceled"` and `subscription_plan="free"`

## 2. Update `handle_webhook` in `backend/app/billing/service.py`

- Accept a database session parameter (or acquire one internally via `get_async_session_factory`)
- After signature verification, dispatch to the appropriate handler function
- Wrap handler calls in try/except to ensure webhook always returns 200 (Stripe retries on failure)
- Add `customer.subscription.deleted` to `_WEBHOOK_ACTIONS` dict

## 3. Update webhook route in `backend/app/billing/routes.py`

- Add `db: AsyncSession = Depends(get_db)` to the `stripe_webhook` endpoint
- Pass the session to `service.handle_webhook`

## 4. Ensure idempotency

- `checkout.session.completed` should be safe to replay: if user already has a `stripe_customer_id`, just update (don't create duplicate)
- `subscription.updated` should be safe to replay: always overwrite with latest state
- `subscription.deleted` should be safe to replay: setting canceled is idempotent

## 5. Write tests

- `backend/tests/test_billing/test_webhook_handlers.py` -- unit tests for each handler with mock event data
- `backend/tests/test_billing/test_webhook_integration.py` -- integration tests hitting the webhook route with constructed payloads (mock Stripe signature verification)

# Acceptance Criteria

- [x] `checkout.session.completed` event creates/links user record with correct `stripe_customer_id` and sets `subscription_status` to `active`
- [x] `customer.subscription.updated` event updates `subscription_plan` and `subscription_status` on the correct user record
- [x] `customer.subscription.deleted` event sets `subscription_status` to `canceled` and `subscription_plan` to `free`

- [x] All tests pass (`pytest backend/tests/test_billing/test_webhook_handlers.py backend/tests/test_billing/test_webhook_integration.py -v`)
