# Objective

There is currently no mechanism to restrict API access based on a user's subscription plan. Any authenticated user can access any endpoint. Add a feature-gating layer that checks the user's subscription entitlements before allowing access to premium features.

This task creates two new modules:
1. A **plan definitions** module that declares plan tiers and their feature access
2. An **entitlements** FastAPI dependency that checks whether the current user's plan grants access to the requested feature

# Background

- Auth is handled by `get_current_user_id` in `backend/app/core/auth.py` which returns a `clerk_user_id` string.
- The Users model from T13.1 stores `subscription_plan` (e.g., `"free"`, `"pro"`, `"enterprise"`) and `subscription_status` (e.g., `"active"`, `"canceled"`).
- The webhook handlers from T13.2 keep `subscription_plan` and `subscription_status` in sync with Stripe.
- Existing routes use `Depends(get_current_user_id)` for auth. The entitlements check should compose with this pattern.
- The items module at `backend/app/items/routes.py` is a good candidate for demonstrating feature gating (e.g., limit number of items for free plan).

# Implementation Plan

## 1. Create plan definitions (`backend/app/billing/plans.py`)

Define plan tiers and feature access as a data structure (not hardcoded in middleware):

```python
# Example structure
PLAN_TIERS = {
    "free": {
        "features": {"items.create", "items.read", "items.update", "items.delete"},
        "limits": {"max_items": 10},
    },
    "pro": {
        "features": {"items.create", "items.read", "items.update", "items.delete",
                      "billing.portal", "export.csv"},
        "limits": {"max_items": 1000},
    },
    "enterprise": {
        "features": {"*"},  # all features
        "limits": {"max_items": -1},  # unlimited
    },
}
```

Include:
- `get_plan_features(plan_name: str) -> set[str]` -- returns feature set for a plan
- `get_plan_limit(plan_name: str, limit_key: str) -> int` -- returns limit value
- `ACTIVE_STATUSES` -- set of statuses that grant access (e.g., `{"active", "trialing"}`)

## 2. Create entitlements dependency (`backend/app/core/entitlements.py`)

A FastAPI dependency factory that checks subscription entitlements:

```python
def require_feature(feature: str) -> Callable:
    """FastAPI dependency that checks the user has access to a feature.

    Usage:
        @router.get("/premium-data", dependencies=[Depends(require_feature("export.csv"))])
        async def get_premium_data(...): ...
    """
```

Logic:
1. Get `clerk_user_id` from auth (compose with `get_current_user_id`)
2. Look up user record from database
3. Check `subscription_status` is in `ACTIVE_STATUSES`
4. Check feature is in the user's plan's feature set
5. If denied, raise `HTTPException(403)` with clear message ("Feature requires Pro plan")

Also provide:
- `require_plan(min_plan: str)` -- dependency that checks minimum plan tier
- `get_user_entitlements(db, clerk_user_id) -> dict` -- returns the user's plan info and available features (useful for frontend)

- The `entitlements.py` file goes in `backend/app/core/` because it is cross-cutting infrastructure, not billing-specific logic. Confirm with the user before creating files there, or place it in `backend/app/billing/entitlements.py` if core is off-limits.
- Keep the plan definitions simple and extensible. Avoid complex class hierarchies; a plain dict or dataclass structure is preferred.
- The `get_user_entitlements` function is designed to be called from a future `/api/v1/users/me/entitlements` endpoint that the frontend can query.
