# Objective

No automated backup strategy exists, which is a SOC2 gap for data availability and recoverability. Implement an automated backup solution with clearly defined and documented RPO (Recovery Point Objective) and RTO (Recovery Time Objective) targets.

The approach depends on the database setup:
- **If Turso (managed)**: Document managed backup capabilities, configure any available backup settings, and define RPO/RTO based on Turso's SLA.
- **If self-hosted SQLite**: Add Litestream for continuous replication or a cron-based backup script with offsite storage.

Target RPO: < 1 hour. Target RTO: < 4 hours.

# Implementation Plan

1. **Determine the database setup**: Check the project's database configuration to identify whether Turso (managed) or self-hosted SQLite is used.
2. **For self-hosted SQLite (primary path)**:
   a. **Create `scripts/backup.sh`**:
      - Accept parameters for database path, backup destination, and retention count.
      - Create timestamped backup copies of the SQLite database file.
      - Use `sqlite3 .backup` command for consistent backups (not raw file copy).
      - Implement retention policy (e.g., keep last 24 hourly backups, last 7 daily backups).
      - Support optional upload to S3-compatible storage.
      - Include error handling and exit codes.
      - Log backup operations with timestamps.
   b. **Add cron schedule** or document how to set up cron for hourly backups.
   c. **Optionally add Litestream** to `docker-compose.yml`:
      - Add Litestream as a sidecar container for continuous WAL replication.
      - Configure replication to an S3-compatible destination.
      - This provides near-zero RPO as an alternative to cron-based backups.
3. **For Turso (managed path)**:
   a. Document Turso's built-in backup and replication features.
   b. Configure any available backup retention settings via Turso CLI or dashboard.
   c. Define RPO/RTO based on Turso's documented SLA.
4. **Create `docs/BACKUP-RECOVERY.md`**:
   - **Backup Strategy**: Which approach is used, frequency, retention policy.
   - **RPO/RTO Targets**: RPO < 1 hour, RTO < 4 hours. Justification for each.
   - **Backup Locations**: Where backups are stored (local, S3, Turso managed).
   - **Restore Procedure**: Step-by-step instructions to restore from backup, including:
     - How to find the latest backup
     - How to verify backup integrity
     - How to restore to the running database
     - How to verify the restored database
   - **Testing Schedule**: How often restore procedures should be tested.
   - **Monitoring**: How to verify backups are running (check logs, file timestamps, alerting on failure).
5. **Update `docs/COMPLIANCE.md`**: Mark the data backup/recovery gap as addressed with a reference to `docs/BACKUP-RECOVERY.md`.
