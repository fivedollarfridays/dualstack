# Objective

The CI pipeline has no deployment step. Add deployment stages to GitHub Actions:
- **Staging**: Auto-deploy on merge to `main`.
- **Production**: Deploy on tag/release creation, with a manual approval gate.

Include environment-level approvals via GitHub Environments, and document the rollback procedure.

# Files

| File | Action |
|------|--------|
| `.github/workflows/deploy.yml` | Create -- deployment workflow with staging and production stages |
| `.github/workflows/ci.yml` | Update -- add deploy trigger or ensure CI passes before deploy |

# Implementation Plan

1. **Create `.github/workflows/deploy.yml`**:
   - **Staging deployment job**:
     - Trigger: `push` to `main` branch (after merge).
     - Environment: `staging` (configure in GitHub repo settings).
     - Steps:
       a. Checkout code
       b. Setup Node.js and install dependencies
       c. Build the application
       d. Deploy to staging environment (placeholder step -- configure for actual hosting provider: Fly.io, Railway, Vercel, etc.)
       e. Run smoke tests against staging URL (optional, can be a follow-up)
       f. Post deployment status to Slack/notification channel (optional)
     - Concurrency: Cancel in-progress staging deploys when a new merge arrives.
   - **Production deployment job**:
     - Trigger: `release` event (published) or `push` of a tag matching `v*`.
     - Environment: `production` with required reviewers (manual approval gate).
     - Steps:
       a. Checkout code at the tag
       b. Setup Node.js and install dependencies
       c. Build the application
       d. Wait for manual approval (handled by GitHub Environment protection rules)
       e. Deploy to production environment
       f. Verify deployment health
     - Concurrency: Do not cancel in-progress production deploys.
2. **Configure GitHub Environments** (document steps, cannot be automated via workflow file):
   - Create `staging` environment in repo settings.
   - Create `production` environment with required reviewers and deployment branch rules (only `main` or tags).
   - Document these setup steps in the workflow file comments or in a deployment doc.
3. **Update `.github/workflows/ci.yml`** if needed:
   - Ensure CI tests pass before deployment by either:
     - Using `workflow_run` trigger in deploy.yml to wait for CI completion, or
     - Adding deployment as a dependent job within the same workflow.
4. **Document rollback procedure**:
   - Add comments in the deploy workflow or a section in `docs/` explaining:
     - How to rollback staging (redeploy previous commit)
     - How to rollback production (redeploy previous tag/release)
     - Emergency rollback steps (manual intervention)
