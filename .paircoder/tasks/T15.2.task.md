# Objective

Configuration is spread across `backend/.env.example`, `frontend/.env.example`, and `monitoring/README.md` (which documents Prometheus, Grafana, and Alertmanager env vars). Developers and operators must hunt through multiple files to understand what environment variables exist, which are required, and what their defaults are. Create a single reference document that consolidates all environment variables across both application stacks and the monitoring stack.

# Files

- `docs/CONFIGURATION.md` (new) -- central configuration reference document

# Implementation Plan

1. Audit all env var sources:
   - `backend/.env.example`: ENVIRONMENT, TURSO_DATABASE_URL, TURSO_AUTH_TOKEN, CLERK_JWKS_URL, STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET, LOG_LEVEL, CORS_ORIGINS, METRICS_API_KEY
   - `frontend/.env.example`: NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY, NEXT_PUBLIC_API_URL, NEXT_PUBLIC_STRIPE_PRO_PRICE_ID, TURSO_DATABASE_URL, TURSO_AUTH_TOKEN, STRIPE_PUBLISHABLE_KEY, ENVIRONMENT, E2E_CLERK_USER_USERNAME, E2E_CLERK_USER_PASSWORD
   - `monitoring/README.md`: GRAFANA_ADMIN_PASSWORD, METRICS_API_KEY, PROMETHEUS_BEARER_TOKEN, PROMETHEUS_BASIC_AUTH_PASSWORD, GF_DATASOURCE_PROM_USER, GF_DATASOURCE_PROM_PASSWORD
2. For each variable, document:
   - Name
   - Which stack/component it belongs to (Backend, Frontend, Monitoring)
   - Description of purpose
   - Required vs. optional (and in which environment: dev, production, or both)
   - Default value (if any)
   - Example value
   - Security notes (e.g., "never commit", "must match across stacks")
3. Organize into sections by component: Backend, Frontend (client-side), Frontend (server-side), Frontend (E2E testing), Monitoring
4. Add a cross-reference section noting variables that must match across stacks (e.g., CLERK_JWKS_URL environment must match NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY environment; METRICS_API_KEY must equal PROMETHEUS_BEARER_TOKEN)
5. Add a quick-start table showing the minimum variables needed for local development
