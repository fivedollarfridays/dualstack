# Objective

No tested restore process or backup monitoring alerts exist -- SOC2 CC6.1 gaps. The COMPLIANCE.md document identifies "Document and test the restore procedure" and "Set up backup monitoring alerts (backup age, backup size anomalies)" as P2 remediation items. This task depends on T14.3 (automated backup strategy) being in place. Create a restore script, validate it end-to-end, add Prometheus alert rules for backup health, and document the full backup/recovery procedure.

# Files

- `scripts/restore.sh` (new) -- database restore script supporting both Turso hosted and local SQLite
- `monitoring/prometheus/alerts/backup.yml` (new) -- Prometheus alert rules for backup age and size anomalies
- `monitoring/prometheus/prometheus.yml` (update, if needed) -- ensure backup alert rules file is loaded
- `docs/BACKUP-RECOVERY.md` (new or update) -- complete backup and recovery procedure documentation

# Implementation Plan

1. Create `scripts/restore.sh`:
   - Accept arguments: `--backup-file <path>` (for local SQLite restore) or `--turso-snapshot <id>` (for Turso hosted restore)
   - Pre-restore checks: verify backup file exists and is valid SQLite, check current database size for sanity comparison
   - Create a safety backup of the current database before restoring (timestamped copy)
   - Perform the restore:
     - Local SQLite: stop the backend service, copy backup file to database path, restart backend
     - Turso hosted: use Turso CLI to restore from snapshot (document the `turso db restore` command)
   - Post-restore validation: verify database is accessible, run a basic query (e.g., count tables, check schema version)
   - Print summary: backup restored from, restore timestamp, validation result
   - Exit non-zero on any failure with descriptive error message
2. Create Prometheus alert rules in `monitoring/prometheus/alerts/backup.yml`:
   - `BackupTooOld`: fires when the last successful backup timestamp is >2 hours old (warning) or >6 hours old (critical)
   - `BackupSizeAnomaly`: fires when backup size deviates >50% from the 7-day average (warning)
   - `BackupJobFailed`: fires when the backup job has failed (uses `background_job_failures_total` metric if backup runs as a background job, or a custom metric)
   - Note: These rules assume a `backup_last_success_timestamp` and `backup_size_bytes` metric are exposed by the backup process. Document the expected metric names.
3. Update `monitoring/prometheus/prometheus.yml` to include the new `alerts/backup.yml` rule file (if not already using a wildcard glob)
4. Create or update `docs/BACKUP-RECOVERY.md`:
   - **Backup Strategy**: Reference T14.3 implementation (automated backups, schedule, retention)
   - **RPO/RTO Targets**: Reference the defined targets from T14.3
   - **Restore Procedure**: Step-by-step guide using `scripts/restore.sh`
     - From local SQLite backup file
     - From Turso hosted snapshot
   - **Restore Testing**: Document the test procedure and results
     - Create a test backup
     - Restore to a separate location
     - Validate data integrity
     - Record test date and outcome
   - **Backup Monitoring**: Describe the Prometheus alerts and what to do when they fire
     - `BackupTooOld`: Check backup job logs, re-run manually if needed
     - `BackupSizeAnomaly`: Investigate data growth or potential corruption
     - `BackupJobFailed`: Check error logs, verify storage access
   - **Disaster Recovery Runbook**: Condensed step-by-step for emergency restore
