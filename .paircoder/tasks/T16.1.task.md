# Objective

No email system exists in the application. Integrate a transactional email provider (Resend recommended for developer experience, with Postmark or SES as alternatives). Create an email service abstraction with templates for: welcome email, password reset (if applicable with Clerk), billing receipt, and generic notification. The service must support async sending to avoid blocking request handlers.

# Files

Source files:
- `backend/app/core/email.py` (new) -- email service class with provider abstraction
- `backend/app/core/email_templates/` (new directory) -- template files for each scenario
- `backend/app/core/email_templates/__init__.py` (new) -- template registry and loader
- `backend/app/core/email_templates/welcome.py` (new) -- welcome email template
- `backend/app/core/email_templates/password_reset.py` (new) -- password reset template
- `backend/app/core/email_templates/billing_receipt.py` (new) -- billing receipt template

- `backend/requirements.txt` -- add `resend` SDK (or chosen provider)

Test files:

- `backend/tests/core/test_email_templates.py` (new) -- tests for template rendering

# Implementation Plan

1. Write tests for the email service interface:
   - `EmailService` initializes with provider config from env vars
   - `send()` accepts `to`, `subject`, `template_name`, `template_data` and returns a message ID
   - `send()` raises `EmailConfigError` when provider is not configured
   - `send_async()` queues email for background sending
2. Implement `email.py` with `EmailService` class wrapping the Resend SDK
3. Write tests for template rendering:
   - `render_template("welcome", {...})` returns subject and HTML body
   - `render_template("password_reset", {...})` includes reset link in output
   - `render_template("billing_receipt", {...})` includes amount and line items
   - `render_template("notification", {...})` renders generic title + message
   - `render_template("unknown")` raises `TemplateNotFoundError`
4. Implement template modules with a registry pattern in `__init__.py`
5. Write integration-style tests with mocked provider:
   - Full flow: render template then send via mocked provider
   - Verify provider SDK called with correct arguments
6. Document required env vars in `.env.example` or project docs

# Acceptance Criteria

- [x] `EmailService` sends emails via the configured provider (Resend SDK or equivalent)
- [x] Templates exist for at least 3 scenarios: welcome, billing receipt, and generic notification
- [x] `send_async()` method sends emails without blocking the calling coroutine
- [x] All email sending tests pass with the provider SDK mocked (no real emails sent in tests)
- [x] Required env vars documented: `RESEND_API_KEY`, `EMAIL_FROM_ADDRESS`, `EMAIL_FROM_NAME`
