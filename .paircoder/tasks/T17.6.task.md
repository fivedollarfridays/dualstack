# Objective

No user analytics are captured. Integrate a privacy-friendly analytics provider (Posthog recommended for its self-hosted option, Mixpanel as an alternative). Track key events: signups, page views, feature usage (item creation, file uploads). The integration must respect the Do Not Track (DNT) header, be configurable via env var (enable/disable), and be compatible with the Content Security Policy (CSP) nonce strategy.

# Files

Frontend source files:
- `frontend/src/lib/analytics.ts` (new) -- analytics service abstraction (init, track, identify, page)
- `frontend/src/components/analytics-provider.tsx` (new) -- React provider that initializes analytics on mount
- `frontend/src/app/layout.tsx` -- wrap app with AnalyticsProvider

Frontend test files:
- `frontend/src/lib/__tests__/analytics.test.ts` (new) -- tests for analytics service
- `frontend/src/components/__tests__/analytics-provider.test.tsx` (new) -- tests for analytics provider

# Implementation Plan

1. Write tests for analytics service:
   - `initAnalytics(config)` initializes the provider with the API key
   - `initAnalytics()` is a no-op when `NEXT_PUBLIC_ANALYTICS_ENABLED` is `"false"` or missing
   - `trackEvent(name, properties)` calls the provider's capture/track method
   - `trackEvent()` is a no-op when analytics is disabled
   - `identifyUser(userId, traits)` calls the provider's identify method
   - `trackPageView(url)` calls the provider's page/capture method
   - `isAnalyticsEnabled()` returns `false` when DNT header is set to "1"
   - `isAnalyticsEnabled()` returns `false` when env var is not set
2. Implement `analytics.ts` with provider abstraction
3. Write tests for AnalyticsProvider:
   - Calls `initAnalytics()` on mount
   - Does not initialize when analytics is disabled
   - Tracks page views on route changes
   - Identifies user when authentication state changes
4. Implement `AnalyticsProvider` component
5. Add AnalyticsProvider to root layout (wrapping children)
6. Define key events to track:
   - `user.signed_up` -- on first sign-in
   - `item.created` -- on successful item creation
   - `item.deleted` -- on item deletion
   - `file.uploaded` -- on successful file upload (if T17.1 is complete)
   - `page.viewed` -- on every route change
7. Add `trackEvent()` calls at appropriate locations in existing components

# Acceptance Criteria

- [ ] Analytics provider initializes with the configured API key from `NEXT_PUBLIC_POSTHOG_KEY` (or equivalent)
- [ ] `trackEvent(name, properties)` sends events to the analytics provider
- [ ] Analytics is completely disabled when `NEXT_PUBLIC_ANALYTICS_ENABLED` is not `"true"`
- [ ] Analytics respects the browser's Do Not Track (DNT) header -- no tracking when DNT is "1"
- [ ] Key events are tracked: signups, page views, item creation
- [ ] `AnalyticsProvider` initializes on mount and tracks page views on route changes
- [ ] Analytics script is compatible with CSP nonce strategy (no inline scripts without nonce)
- [ ] All analytics calls are no-ops when disabled (no errors, no network requests)
- [ ] All tests pass
