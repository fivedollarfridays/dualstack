# Objective

Fix CRITICAL security vulnerability: WebSocket JWT authentication bypasses signature verification (`verify_signature: False`), allowing any forged JWT to authenticate as any user.

# Files to Update

| File | Change |
|------|--------|
| `backend/app/core/ws_auth.py` | Rewrite `_verify_token` to verify JWT signature via JWKS |
| `backend/tests/core/test_ws_auth.py` | Add tests for signature verification, forged/expired tokens |

# Implementation Plan

1. Write failing tests first:
   - Test that a forged JWT (signed with wrong key) is rejected
   - Test that an expired JWT is rejected
   - Test that a valid JWT with correct signature returns user_id
   - Test that a JWT missing `sub` claim is rejected
2. Rewrite `_verify_token` in `ws_auth.py`:
   - Use `jwt.PyJWKClient(jwks_url)` to fetch Clerk's public key
   - Use `pyjwt.decode(token, key, algorithms=["RS256"])` with signature verification
   - Cache the `PyJWKClient` instance (similar to `_clerk_auth_cache` in `auth.py`)
   - Remove `verify_signature: False` entirely
3. Verify dev mode path still works (no JWKS URL → trust user_id query param)
4. Run all ws_auth tests

# Acceptance Criteria

- [x] `_verify_token` verifies JWT signature using Clerk JWKS public key
- [x] `verify_signature: False` removed completely
- [x] Dev mode still works (no JWKS URL → trust `user_id` query param)
- [x] Production mode rejects forged/expired JWTs
- [x] Tests cover: valid token, forged token, expired token, missing sub claim
- [x] All existing ws_auth tests pass
