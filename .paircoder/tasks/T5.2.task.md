# Objective

The billing integration has multiple issues: frontend checkout sends an incomplete payload (will 422), the portal endpoint accepts arbitrary customer IDs (IDOR), there's no backend users table for subscription storage, and pricing is inconsistent between the stripe sync script and billing page. Covers issues C4, C5, C6, M6, M18.

# Implementation Plan

## 1. Fix frontend checkout payload (C4)

Update `frontend/src/lib/api/billing.ts` `createCheckout()` to include `success_url` and `cancel_url` from `window.location.origin`. Update the billing page to pass these URLs.

## 2. Gate the portal endpoint (C5)

The portal endpoint accepts `customer_id` from the caller, creating an IDOR. Since there's no user->customer mapping table, the endpoint should return a 501 Not Implemented with a clear message explaining the buyer needs to implement user->customer mapping first. Keep the service code and schema but gate the route.

## 3. Document the users table gap (C6)

Add a clear section in the billing service.py and in the README explaining that the billing pipeline requires a users/customers table to be complete. The webhook handler should log what it receives but document that storage is the buyer's responsibility.

## 4. Add NEXT_PUBLIC_STRIPE_PRO_PRICE_ID to frontend .env.example (M6)

Add the env var with a placeholder value that passes backend validation (e.g., `price_test123`).

## 5. Fix pricing consistency (M18)

Update `scripts/stripe_sync.py` to match the billing page pricing (Free/$0, Pro/$10) or make the script configurable and document that buyers should customize it.

## 6. Update tests

- Frontend: Fix `billing.test.ts` to test the updated payload with success_url and cancel_url
- Frontend: Fix `billing-page.test.tsx` to match updated checkout call
- Backend: Add test for portal endpoint returning 501
