# Objective

The database integration has three issues: Turso is advertised but silently falls back to SQLite, no Drizzle migration files exist, and the frontend/backend schemas use incompatible timestamp formats. Covers issues H2, H3, H4.

# Implementation Plan

## 1. Document Turso limitation honestly (H2)

Update `backend/app/core/database.py` docstring and README to clearly state:
- Backend uses SQLite locally for development
- Turso (libsql://) URLs are detected but fall back to local SQLite because SQLAlchemy lacks a native libsql driver
- For production Turso, buyers need to integrate `sqlalchemy-libsql` or use the HTTP API
- This is a documented limitation, not a bug

Update README tech stack table: change "Turso (LibSQL) / SQLite" to "SQLite (Turso-ready with configuration)"

## 2. Generate Drizzle migration files (H3)

Run `cd frontend && npx drizzle-kit generate` to create the initial migration files. Verify they exist in `frontend/drizzle/`. Update README quickstart if needed.

## 3. Align frontend/backend schemas (H4)

Option A (recommended): Update the frontend Drizzle schema to match backend conventions:
- Change `integer('created_at', { mode: 'timestamp' })` to `text('created_at')` with `(CURRENT_TIMESTAMP)` default to match backend SQLAlchemy
- Remove the foreign key to `users.id` from items schema (backend has no users table)
- Keep the users schema as-is but add a comment noting it's frontend-only until the backend adds a users model

Regenerate Drizzle migrations after schema changes.

### Files to modify
