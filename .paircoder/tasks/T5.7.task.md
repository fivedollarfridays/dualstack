# Objective

Polish the frontend UX so buyers see a functional, professional scaffold they can extend. Covers issues M4, M5, M7, M14, M15.

# Implementation Plan

## 1. Wire dashboard page to API (M4)

Replace hardcoded "--" values with actual data:
- Item count: call `GET /api/v1/items` and show count
- Subscription: show "Free" (static for now, with TODO for when users table exists)
- Remove "credits" metric (not part of the backend)

## 2. Make settings page functional (M5)

- Add a "Manage Account" button that links to Clerk's user profile URL
- Add a link to the billing page for subscription management
- Remove the non-functional "Danger Zone" section or add a working "Delete Account" that calls Clerk's delete API

## 3. Parse error details from backend responses (M7)

Update `frontend/src/lib/api/shared.ts` `handleResponse()` to read the response body JSON and extract the error message from the backend's `{ error: { message: "..." } }` structure. Fall back to status text if parsing fails.

## 4. Add mobile navigation (M14)

Replace `{/* Mobile nav placeholder */}` with a hamburger menu that opens a drawer/sheet containing the same sidebar links. Use existing shadcn/ui Sheet component or a simple toggle.

## 5. Add pagination controls (M15)

Add "Previous" / "Next" buttons to the items page. Wire to the existing `page` and `limit` parameters in `useItems`. Show current page and disable buttons at boundaries.

## 6. Write tests for all changes

- Dashboard: test that item count is fetched and displayed
- Settings: test that Clerk profile link renders
- Shared: test that error details are parsed from response body
- Mobile nav: test that hamburger menu toggles navigation
- Pagination: test that prev/next buttons work
