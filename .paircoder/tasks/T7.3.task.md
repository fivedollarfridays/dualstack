# Objective

Fix rate limiter to use `X-Forwarded-For` header when behind a reverse proxy (SEC-004). Currently `rate_limit.py` uses `get_remote_address` which returns the direct TCP peer IP. Behind Render or any reverse proxy, all users share one rate-limit bucket.

# Files to Update

| File | Change |
|------|--------|
| `backend/app/core/rate_limit.py` | Replace `get_remote_address` with a custom key function that reads `X-Forwarded-For` |
| `backend/tests/core/test_rate_limit.py` | Add tests for proxy-aware IP extraction |

# Implementation Plan

1. Write failing tests:
   - extracts client IP from `X-Forwarded-For` header (first IP in chain)
   - falls back to `request.client.host` when no `X-Forwarded-For`
   - handles empty or malformed `X-Forwarded-For` gracefully
2. Implement `get_client_ip(request: Request) -> str`
3. Replace `key_func=get_remote_address` with `key_func=get_client_ip`

# Acceptance Criteria

- [x] Rate limiter uses `X-Forwarded-For` header when present
- [x] Falls back to direct peer IP when header is absent
- [x] First IP in chain is used (leftmost = real client)
- [x] All tests pass
- [x] Architecture checks pass
