# Objective

Close two findings: catch Stripe webhook signature verification failures with proper error responses and audit logging (NEW-003), and sanitize FastAPI's default 422 validation error responses to prevent schema leakage (NEW-004).

# Files to Update

| File | Change |
|------|--------|
| `backend/app/billing/service.py` | Catch `stripe.error.SignatureVerificationError` and `ValueError` in `handle_webhook()`, raise specific errors |
| `backend/app/core/exception_handlers.py` | Add `RequestValidationError` handler that returns sanitized 422 response |
| `backend/app/core/audit.py` | May need to accept optional `outcome` parameter for failure events |
| `backend/tests/billing/test_service.py` | Add tests for webhook signature failure handling |
| `backend/tests/core/test_exception_handlers.py` | Add test for sanitized 422 response |

# Implementation Plan

1. In `billing/service.py`, wrap `stripe.Webhook.construct_event()` in try/except catching `stripe.error.SignatureVerificationError` and `ValueError`; raise `AuthenticationError` for signature failures; call `log_audit_event` for the security event
2. In `exception_handlers.py`, register a `RequestValidationError` handler that returns `{"error": {"code": "VALIDATION_ERROR", "message": "Invalid request"}}` with status 422
3. Write tests: webhook with invalid signature returns 400/401 (not 500); invalid request body returns sanitized 422 without field details

# Acceptance Criteria

- [x] Stripe webhook with invalid signature returns 400 or 401, not 500
- [x] Invalid webhook payload (ValueError) returns 400, not 500
- [x] Webhook signature failures emit an audit log event
- [x] 422 validation error responses do not include field names, `loc`, or `type` details
- [x] 422 response body matches format `{"error": {"code": "VALIDATION_ERROR", "message": "Invalid request"}}`
- [x] All tests pass
- [x] Architecture checks pass
