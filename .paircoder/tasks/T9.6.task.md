# Objective

Bound the Clerk auth cache to prevent unbounded memory growth, and replace `window.confirm()` with a proper confirmation component for destructive actions.

# Findings Addressed

- **AUDIT-011** (Low): `_clerk_auth_cache` is unbounded global dict
- **AUDIT-015** (Low): `window.confirm()` for destructive actions can be suppressed

# Files to Update

| File | Change |
|------|--------|
| `app/core/auth.py` | Add max size check to `_clerk_auth_cache` (bounded by JWKS URLs, typically 1) |
| `frontend/src/app/items/[id]/page.tsx` | Replace `window.confirm()` with confirmation dialog component |
| `frontend/src/components/ui/confirm-dialog.tsx` | Create reusable confirmation dialog |
| `frontend/src/components/ui/confirm-dialog.test.tsx` | Tests for confirmation dialog |
| `tests/core/test_auth.py` | Test cache bounds behavior |

# Implementation Plan

1. Add a max-size guard to `_clerk_auth_cache` (e.g., max 10 entries, clear oldest on overflow) -- in practice only 1 JWKS URL per deployment, but defense-in-depth
2. Create a `ConfirmDialog` component using native HTML `<dialog>` element
3. Replace `window.confirm()` in item deletion with the new component
4. Write tests for both changes

# Acceptance Criteria

- [x] `_clerk_auth_cache` has a maximum size limit with eviction
- [x] Cache bound is tested
- [x] `window.confirm()` is no longer used for destructive actions
- [x] Confirmation dialog is accessible and cannot be programmatically suppressed
- [x] Existing auth and frontend tests pass
